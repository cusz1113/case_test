# 需求拆分技能 (Requirement Splitter)

## 1. 技能定位

本技能专注于**阶段2：需求梳理**工作，负责：

1. 解析 md 格式的需求文档
2. 按章节结构识别功能模块
3. 拆分需求单元并统一编号
4. 提取需求关联的图片
5. 生成 JSON 格式的结构化需求文档

**不包含**：测试点提取（由其他技能负责）

***

## 0. JSON 文件格式严格要求（最高优先级）

**生成任何 JSON 文件时必须严格遵守以下规则，违反即为不可接受的错误：**

1. **必须使用 `json.dumps()` 写入文件**：禁止手动拼接 JSON 字符串。所有 JSON 输出必须通过 Python 的 `json.dump(data, f, ensure_ascii=False, indent=2)` 或等价方式生成，确保格式100%有效。

2. **中文引号规则**：JSON 字符串值内部的中文双引号必须使用全角引号 `“`（"）和 `”`（"），严禁使用 ASCII 双引号 `"`（"）出现在字符串值内部。例如：
   - 错误：`"description": "用户点击"确定"按钮"`
   - 正确：`"description": "用户点击“确定”按钮"`

3. **写入后必须验证**：写入 JSON 文件后，必须立即用 `json.load()` 重新读取验证。验证失败则必须修复后重新写入，不得跳过。

4. **常见错误清单**（写入前逐项检查）：
   - 字符串值内出现未转义的 ASCII `"`（最常见错误）
   - 字符串值内出现未转义的反斜杠 `\`
   - 字符串值内出现未转义的换行符（应使用 `\n`）
   - 尾随逗号（数组/对象最后一个元素后多余的 `,`）
   - 单引号用作字符串界定符（JSON 只支持双引号）

> **违反以上任何一条规则的 JSON 文件视为不合格产物，必须重新生成。**

## 3. JSON 需求文档格式

### 3.0 可测试需求的核心要求（重要）

**每个需求必须包含以下要素，否则不能作为独立需求提取：**

| 要素        | 说明          | 必须性 |
| --------- | ----------- | --- |
| **触发条件**  | 什么情况下触发这个功能 | 必须  |
| **输入**    | 需要什么数据/参数   | 必须  |
| **动作/操作** | 用户或系统做什么操作  | 必须  |
| **预期输出**  | 期望的结果是什么    | 必须  |
| **验证点**   | 如何验证功能正确    | 必须  |

> **如果一个需求无法回答上述5个问题，则不是可测试的需求，不应提取。**

### 3.1 单个需求单元结构

```json
{
  "requirement_id": "REQ-CHX-001",
  "requirement_name": "新增{业务对象}信息",
  "type": "功能需求 (FR)",
  "module": "{业务对象}管理",
  "chapter": X,
  "source_page": "第 XX 页",

  "description": "{角色}可以在{场景}下新增{业务对象}信息",

  "testable_elements": {
    "trigger": "{触发条件}",
    "inputs": ["{输入参数1}", "{输入参数2}"],
    "action": "{执行动作}",
    "expected_output": "{预期输出}",
    "verification_points": [
      "{验证点1}",
      "{验证点2}"
    ]
  },

  "detailed_description": "【用例 UC-XXX-001】\n\n{完整的需求原文描述}",
  "acceptance_criteria": [
    "{验收标准1}",
    "{验收标准2}"
  ],

  "related_images": [
    {
      "filename": "xxx.jpg",
      "description": "{图片描述}",
      "path": "images/xxx.jpg"
    }
  ],
  "source_paragraphs": ["Para-XXX-001"]
}
```

### 3.2 章节需求 JSON 结构

```json
{
  "chapter": {
    "chapter_number": 3,
    "chapter_title": "XXXXXX",
    "total_requirements": 25
  },
  "modules": [
    {
      "module_name": "XXXXXX",
      "requirement_count": 5,
      "requirements": [
        // 每个 requirement 包含完整 detailed_description
      ]
    },
    {
      "module_name": "XXXXX",
      "requirement_count": 8,
      "requirements": [
        // 每个 requirement 包含完整 detailed_description
      ]
    }
  ]
}
```

## 4. 需求拆分规则

### 4.1 章节识别规则

1. 识别文档中的**内容一级标题**（`#`）作为章(如果该章节下面没有具体的需求则跳过)
2. 识别文档中的功能分类 作为功能模块的名称
3. 每个一级标题下的内容属于同一章

### 4.2 需求单元识别规则

#### 4.2.1 需求识别原则（重要）

**必须提取的功能性需求：**

- 有明确动作动词的需求（如：新增、删除、修改、查询、展示、使用、发放、导出等）
- 可以设计具体测试用例的需求
- 有明确输入、输出、触发条件的需求
- 【用例UC-xxx】描述的功能点
- 【需求场景】中描述的具体功能

**不应提取的非功能性内容（跳过）：**

- 背景描述、业务背景、项目背景
- 概念说明、术语定义、业务前提与假设
- 章节/小节概述性文字
- 业务流程概述（非具体功能点）
- 通用说明、注意事项（无具体功能操作）
- 列表形式的属性/字段说明（无增删改查动作）
- "系统应支持..."开头但无具体操作描述的概括性内容

**判断标准：**

> 如果一个需求**不能设计出具体的测试用例**（没有明确的操作动作、没有可验证的输入输出），则不应作为独立需求提取。

#### 4.2.2 识别关键词信号

| 应提取（有关键动作） | 不应提取（仅有描述性文字） |
| ---------- | ------------- |
| 新增、创建、增加   | 概述、说明、指、表示    |
| 删除、移除、注销   | 背景、概念、定义、前提   |
| 修改、更新、变更   | 适用于、一般、用于、属于  |
| 查询、检索、搜索   | 包括、含有、具备、拥有   |
| 展示、显示、呈现   | 满足、实现、提供、支持   |
| 使用、应用、调用   | 通过、根据、按照、随着   |

#### 4.2.3 字段说明

| 字段                    | 说明                   | 来源                   | 完整性要求                 |
| --------------------- | -------------------- | -------------------- | --------------------- |
| requirement\_id       | 唯一标识，如 `REQ-CH3-001` | 自动生成                 | -                     |
| requirement\_name     | 简洁的功能描述              | 从文本中提取               | -                     |
| type                  | 功能需求/性能需求/安全需求等      | 根据内容判断               | -                     |
| module                | 所属的功能模块              | 根据内容判断               | -                     |
| source\_page          | 在原文档中的页码位置           | 从文本中解析               | -                     |
| description           | 需求简述（一句话概括）          | 从文本中提取               | -                     |
| detailed\_description | **完整原始需求描述**         | 保留原文所有内容             | **必须完整保留原文所有细节，不得截断** |
| acceptance\_criteria  | 验收标准列表               | 从文本中提取               | -                     |
| related\_images       | 关联的图片列表              | 解析 `![](images/xxx)` | -                     |
| source\_paragraphs    | 源段落锚点列表              | 根据行号估算               | -                     |

> **重要**：`detailed_description` 字段必须完整保留从原文档中提取的需求内容，包括：
>
> - 所有功能点描述
> - 所有业务规则
> - 所有条件限制
> - 所有数据字段要求
> - 原文中的所有细节
>
> 不得省略、摘要或简化原文内容。

### 4.3 需求拆分稳定性规则（重要）

为保证多次运行结果一致性，必须遵守以下规则：

#### 4.3.1 顺序保证

- **严格按照原文段落顺序提取需求**，不得打乱顺序
- 需求ID编号必须与原文段落出现顺序一致
- 不允许跨段落合并多个需求为一个

#### 4.3.2 拆分粒度标准

- **一个【用例UC-xxx】= 一个需求单元**
- **一个【需求场景】= 一个需求单元**
- **一个独立的功能点标题 = 一个需求单元**
- 不允许将多个独立需求合并为一个
- 不允许将一个完整需求拆分多个

#### 4.3.3 模块归属映射规则

**通用模块识别规则**（按优先级执行）：

| 优先级 | 识别规则                | 模块归属         |
| --- | ------------------- | ------------ |
| 1   | 包含"新增/创建/增加/增"关键词   | 后台管理         |
| 2   | 包含"删除/移除/注销/删"关键词   | 后台管理         |
| 3   | 包含"修改/更新/变更/改"关键词   | 后台管理         |
| 4   | 包含"查询/检索/搜索/查"关键词   | 后台管理         |
| 5   | 包含"展示/显示/呈现"关键词     | 前台/用户界面      |
| 6   | 包含"使用/应用/调用"关键词     | 前台/用户界面      |
| 7   | 包含"查看/浏览"关键词        | 前台/用户界面      |
| 8   | 包含"导出/报表/统计"关键词     | 报表功能         |
| 9   | 包含"新增.\*类型/创建.\*类型" | {业务}\_类型管理   |
| 10  | 以上都不匹配              | 功能模块（从功能名推断） |

**模块名称生成规则**：

```
当识别到业务关键词时，模块名 = {业务关键词}+"管理"
示例：
- 新增代金券类型 → 模块：代金券类型管理
- 新增客户信息 → 模块：客户信息管理
- 新增订单 → 模块：订单管理
```

> **强制规则**：模块归属按上述优先级规则自动判断，不得硬编码特定业务术语

#### 4.3.4 需求名称生成规则

```
格式：{动作}+{业务对象}
示例：
- 新增用户信息 → "新增用户信息"
- 查询订单 → "查询订单"
- 修改密码 → "修改密码"
- 删除商品 → "删除商品"
```

#### 4.3.5 描述提取规则

**description格式（强制）**：

```
格式：[角色]可以在[场景]下[执行动作]来[实现目标]
示例：管理员可以在用户管理页面新增用户信息
```

**description不满足测试需求时的处理**：

- 如果原文描述不包含完整的主语+动作+宾语，补全但不改变原意
- 如果描述过于笼统（如"系统应支持..."），需细化具体操作
- 如果描述是业务规则而非功能操作，不提取为独立需求

**detailed\_description**：保留原文所有内容，不删减。必须包含：

- 完整的操作步骤
- 具体的业务规则
- 所有的条件限制
- 字段定义和取值范围

#### 4.3.6 testable\_elements强制要求

每个需求**必须包含**testable\_elements字段，否则该需求不可用：

```json
"testable_elements": {
  "trigger": "触发条件：什么操作或事件触发了这个功能",
  "inputs": ["输入1", "输入2"],
  "action": "执行动作：用户或系统做了什么",
  "expected_output": "预期输出：期望的结果是什么",
  "verification_points": ["验证点1", "验证点2"]
}
```

**如果无法提取testable\_elements，该需求应标记为"非功能需求"或跳过**。

#### 4.3.6 一致性校验

完成拆分后必须校验：

1. 需求ID连续不间断（如：REQ-CH3-001, REQ-CH3-002...）
2. 每个模块的requirement\_count与实际requirements数组长度一致
3. 章节total\_requirements等于所有模块requirement\_count之和

### 4.4 需求编号规则

```
REQ-{章}-{序号:3位}
例：REQ-CH3-001  表示第3章的第1个需求
```

## 5. 图片关联规则

### 5.1 图片识别

1. 解析 md 文档中的图片引用语法：`![](images/xxx.jpg)`
2. 记录图片的原始路径和文件名
3. 判断图片属于哪个需求单元（根据上下文段落判断）

### 5.2 图片归属判定

- 图片与其**前一个段落**中的需求关联
- 如果前一个段落是列表项，与列表项中的需求关联
- 图片复制到对应章节的 `images/` 目录下

### 5.3 图片复制操作（必须执行）

在生成每个章节的 `requirements.json` 时，必须同时执行以下图片复制操作：

1. **创建图片目录**：每个章节目录下必须创建 `images/` 子目录
2. **提取图片列表**：从该章节的 `requirements.json` 中提取所有 `related_images` 的 `filename`
3. **复制图片**：从原始图片目录 `02_extracted/images/` 复制到对应章节的 `images/` 目录
4. **记录复制结果**：统计复制的图片数量

### 5.4 图片复制命令示例

```bash
# 为每个章节创建 images 目录
mkdir -p "output/{项目名称}/03_requirements/01_第{章}章_{章节名}/images"

# 复制该章节需要的图片（假设已提取图片列表）
cp "output/{项目名称}/02_extracted/images/{图片文件名}.jpg" \
   "output/{项目名称}/03_requirements/01_第{章}章_{章节名}/images/"
```

### 5.5 JSON 中的图片字段

```json
"related_images": [
  {
    "filename": "原文件名.jpg",
    "description": "图片描述（根据上下文推断）",
    "path": "images/原文件名.jpg"
  }
]
```

## 6. 验收标准提取规则

### 6.1 识别关键词

验收标准通常出现在以下关键词后：

- "验收标准"
- "完成条件"
- "验收条件"
- "验证条件"
- "上线标准"
- "功能要求"
- "需求"

### 6.2 验收标准格式要求

**好的验收标准示例**：

```
✅ 管理员新增用户时，用户名不能为空
✅ 输入有效的查询条件，系统返回对应的信息列表
✅ 删除已有关联数据的记录时，系统提示无法删除
✅ 当输入数据格式错误时，系统提示相应的错误信息
```

**不合格的验收标准示例**：

```
❌ 系统应支持用户管理功能（太笼统）
❌ 提供良好的用户体验（无法验证）
❌ 符合相关法规要求（无法验证）
```

### 6.3 验收标准提取规则

1. 查找包含验收关键词的段落
2. 提取**具体可测量的条款**（不是概述性描述）
3. 每个验收标准必须包含：
   - 具体的条件（如：输入、状态、前提）
   - 可验证的结果（如：显示、返回、提示）
4. 如果原文没有明确的验收标准，从detailed\_description中**反向推导**可测试的验收标准

### 6.4 验收标准JSON格式

```json
"acceptance_criteria": [
  "管理员可以新增用户信息，保存后列表显示新增的用户",
  "用户名为空时，系统提示'用户名不能为空'",
  "已有关联数据的用户不可删除",
  "用户状态变更后列表中状态同步更新"
]
```

### 6.5 从业务规则推导验收标准

如果原文只描述了业务规则（如计算规则），需推导出可测试的验收点：

```
原文：订单金额满100元可减免10元

推导验收标准：
- 订单金额100元，系统计算并减免10元
- 订单金额99元，系统不进行减免
- 订单金额200元，系统计算并减免20元（每100元减10元）
```

## 9. 执行流程

```
1. 解析输入参数
   ├── 验证 input_doc 是否存在
   └── 验证 output_dir 是否有效

2. 创建目录结构
   ├── 创建 output/{项目名称}/ 目录
   ├── 创建 03_requirements/ 目录
   └── 为每个章节创建章节目录和 images/ 子目录

3. 读取需求文档
   ├── 按行读取 md 文档
   ├── 识别章节结构（一级/二级标题）
   └── 提取段落内容

4. 章节遍历
   └── 对每个章节执行：
       ├── 识别功能模块
       ├── 在模块内识别功能需求单元
       ├── 提取验收标准
       ├── 关联图片（解析 markdown 中的图片引用）
       ├── 生成 JSON 需求文档
       └── 【重要】验证每个需求的testable_elements完整性

5. 图片处理（每个章节完成后必须执行）
   ├── 创建章节的 images/ 目录
   ├── 提取该章节 requirements.json 中的所有图片文件名
   ├── 从 02_extracted/images/ 复制图片到章节的 images/ 目录
   └── 验证图片复制完成

6. 一致性校验（每个章节完成后必须执行）
   ├── 校验需求ID连续不间断
   ├── 校验模块requirement_count与实际数组长度一致
   ├── 校验章节total_requirements等于所有模块之和
   ├── 【重要】校验每个需求的testable_elements字段完整
   └── 如有不一致，报错并列出具体问题

7. 输出报告
   ├── 生成各章节 requirements.json
   ├── 生成 README.md（章节索引）
   └── 输出执行报告
```

**可测试性验证（必须在步骤4执行）**：

```
对于每个识别出的需求单元：
1. 检查是否包含 trigger、inputs、action、expected_output
2. 如果缺少任一要素：
   - 尝试从detailed_description中推导
   - 如果无法推导，标记type为"非功能需求(NFR)"或跳过
3. 确保description格式为"[角色]可以[动作]..."
```

## 10. 错误处理

| 错误情况                     | 处理方式                                                         |
| ------------------------ | ------------------------------------------------------------ |
| 输入文档不存在                  | 报错退出，提示用户检查路径                                                |
| 文档格式异常                   | 跳过异常段落，记录日志                                                  |
| 图片文件缺失                   | 在 JSON 中标注 `image_missing: true`，继续处理                        |
| 图片复制失败                   | 记录警告，继续处理；不影响需求文档生成                                          |
| 验收标准无法提取                 | 从detailed\_description反向推导，仍无法推导则标注`acceptance_criteria: []` |
| 一致性校验失败                  | 报错退出，列出具体不一致项（如ID不连续、计数不匹配）                                  |
| **可测试性验证失败**             | 标记type为`非功能需求(NFR)`或跳过该需求，继续处理其他需求                           |
| **testable\_elements缺失** | 尝试从description推导，如无法推导则标记需求类型并添加note说明                       |

## 11. 需求识别示例

### 11.1 应提取的需求（功能性）

**示例1：【用例 UC-xxx】**

```markdown
新增用户信息

管理员可以在用户管理页面新增用户信息，填写用户名、密码、手机号、邮箱等字段后保存。

【用户属性】
- 用户名：唯一标识
- 手机号：11位数字
- 邮箱：有效的邮箱格式
...
```

→ **提取为需求**："新增用户信息"，模块："用户信息管理"

**示例2：**

```markdown
查询订单列表

输入订单状态、日期范围等条件，查询满足条件的订单信息列表。
```

→ **提取为需求**："查询订单列表"，模块："订单管理"

### 11.2 不应提取的内容（背景/描述性）

**示例1：背景描述**

```markdown
{业务名称}是公司核心系统，用于管理日常业务运营。
系统支持多种业务类型的处理，提供完善的功能模块。
```

→ **不提取**：这是背景概述，不是具体功能操作

**示例2：业务前提**

```markdown
【业务前提与假设】
1. 用户必须已登录系统
2. 操作员需要具有相应权限
3. 数据必须符合格式要求
```

→ **不提取**：这是业务规则说明，不是功能需求

**示例3：概念定义**

```markdown
通过后台管理系统，对{业务对象}进行定义、新增、发放和管理，
并关联至相对应的活动进行人工处理或按照所设定的触发条件自动处理。
```

→ **不提取**：这是系统功能概述，不是可测试的需求点

**示例4：列表属性说明**

```markdown
【{业务对象}属性】
- 代码：唯一标识
- 名称：显示名称
- 状态：有效/无效
- 创建时间：日期时间
```

→ **不提取**：这是字段属性定义，不是功能操作。若某个属性需要单独的增删改查功能，才单独提取。

### 11.3 判断流程

```
遇到一个段落
    │
    ▼
是否有明确动作动词？（新增/删除/修改/查询/展示/使用/导入/导出）
    │
    ├── 否 → 检查是否是业务规则/背景描述/概念定义
    │         └── 是 → 不提取为独立需求，跳过
    │
    └── 是 → 是否可设计具体测试用例？（有输入、输出、触发条件）
              │
              ├── 是 → 提取为需求
              └── 否 → 归类为业务规则，关联到最近的功能需求
```

### 11.4 不可测试需求的处理

**以下内容不应提取为独立需求：**

| 类型          | 处理方式           |
| ----------- | -------------- |
| 背景描述        | 跳过，不提取         |
| 概念/术语定义     | 跳过，不提取         |
| 业务前提/假设     | 跳过，不提取         |
| 概述性文字       | 跳过，不提取         |
| 字段属性说明（无操作） | 跳过，不提取         |
| 业务流程概述      | 关联到具体功能需求作为上下文 |
| 通用说明/注意事项   | 跳过，不提取         |

**如果某段文字描述的是"系统应该支持X"但没有具体操作步骤**：

1. 检查是否有具体的UC用例标题（如"UC-xxx"）
2. 如果有，按照UC用例提取为独立需求
3. 如果没有，将内容作为背景信息归并到最近的功能需求中

### 11.5 特殊需求类型标记

**对于无法完全跳过但又不完整的内容，使用type字段标记：**

```json
{
  "type": "非功能需求 (NFR)",
  "note": "仅包含业务规则，无法提取完整的测试要素"
}
```

可用的type值：

- `功能需求 (FR)` - 可直接测试的功能
- `非功能需求 (NFR)` - 性能、安全等无法直接测试的需求
- `业务规则 (BR)` - 业务规则说明，需关联到具体功能
- `数据需求 (DR)` - 数据字段定义，需关联到具体功能
